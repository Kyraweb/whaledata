System architecture

The platform follows a three-tier architecture: a PostGIS database, a FastAPI REST API, and a Vue 3 frontend. All services run as independent Docker containers managed by Coolify. A separate jobs service runs six data sync scripts on cron schedules.

The monorepo structure separates concerns cleanly: api/ for the FastAPI backend and admin panel, map/ for the Vue frontend, jobs/ for the data sync workers. Each has its own Dockerfile and can be deployed independently. Coolify's webhook-triggered deployment means a push to GitHub rebuilds only the service whose directory changed.

Database — PostgreSQL + PostGIS 12 tables

PostgreSQL was the obvious choice — free, open source, extraordinarily capable, and the de facto standard for geospatial data when combined with PostGIS. The spatial extension stores every location as a GEOGRAPHY(POINT, 4326) — a geometric point on the surface of a sphere using WGS84 coordinates (the same datum as GPS).

This means spatial queries are first-class operations. Finding all sightings within 500km of a user's location is a single SQL call using ST_DWithin(location, point, 500000) — no client-side filtering required.

Schema — sightings table (simplified)
CREATE TABLE sightings (
  id              SERIAL PRIMARY KEY,
  common_name     VARCHAR(255),
  location        GEOGRAPHY(POINT, 4326) NOT NULL,
  sighted_on      DATE,
  source          VARCHAR(50)  -- gbif / obis
  source_id       VARCHAR(255),
  UNIQUE (source, source_id)  -- deduplication
);

CREATE INDEX sightings_location_idx
  ON sightings USING GIST (location);

The UNIQUE (source, source_id) constraint is the deduplication mechanism. Every sync script uses ON CONFLICT DO NOTHING — inserting new records and silently skipping duplicates. This means sync jobs can run as many times as needed without creating duplicate data, and partial syncs can resume safely after failures.

API — FastAPI Python 3.11

FastAPI was chosen for its automatic OpenAPI documentation generation, excellent performance for a Python framework, and clean dependency injection pattern. The API handles all data serving, filtering, and the admin panel.

The admin panel is built with Jinja2 server-rendered templates rather than a Vue frontend. This was a deliberate choice — an admin interface for internal use doesn't need client-side reactivity, and server-rendered HTML is simpler to maintain. HTTP Basic Auth protects all admin routes.

All filtering happens server-side. When a user selects Humpback whale and sets a year range, the API returns only the matching records — not all 25,000 with client-side filtering. This is essential for mobile performance where data transfer is the bottleneck.

Frontend — Vue 3 + MapTiler SDK WebGL

Vue 3 with Vite handles the interface and state management. The map itself is rendered by the MapTiler SDK — a library built on MapLibre GL JS v5 that uses WebGL to render the globe directly on the GPU.

The 25,000 sighting dots are not individual DOM elements. They are a single GPU layer — a set of instructions sent to the graphics card that tells it to draw circles at specific coordinates, in specific colours, at a specific size that varies with zoom level. This is why the globe renders smoothly on a mobile phone: the GPU is doing the work, not the CPU.

Map — GPU circle layer
map.addLayer({
  id: 'sightings-dot',
  type: 'circle',
  source: 'sightings',
  paint: {
    'circle-radius': [
      'interpolate', ['linear'], ['zoom'],
      1, 3,   // zoom 1 → 3px radius
      5, 5,   // zoom 5 → 5px radius
      10, 8   // zoom 10 → 8px radius
    ],
    'circle-color': ['get', 'color']
  }
})

The animated migration route lines use a frame-stepping dash array technique — cycling through 14 pre-computed dash configurations at 80ms intervals creates the illusion of motion along the line. This avoids the performance cost of animating individual points along a path.

Deployment — Coolify + Docker Self-hosted

Coolify is a self-hosted deployment platform that runs on the VPS and manages Docker containers, SSL certificates via Let's Encrypt, GitHub webhooks for auto-deployment, and scheduled tasks for cron jobs. It handles in a web interface what would otherwise require manual Nginx configuration, Certbot setup, and cron management.

Why Coolify over a managed platform?

Managed platforms (Vercel, Render, Railway) work well for simple deployments but become expensive or limiting for multi-service applications with persistent databases, background jobs, and custom domains. Coolify on a $12/month Debian VPS provides full control at a fraction of the cost — and the PostGIS database requirement ruled out most serverless options anyway.

The six sync jobs run as Coolify Scheduled Tasks — cron expressions attached to the jobs service container. GBIF and OBIS run weekly, strandings and acoustics monthly, iNaturalist weekly, historical records yearly. The jobs service runs continuously but the scripts only execute when triggered by the scheduler.
