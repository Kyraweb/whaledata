Reading the tail: The p95 for the full sightings endpoint is 11 seconds — meaning 95% of requests complete within 11s under 50 concurrent users. For a globe that loads once per session rather than per interaction, this is acceptable. Filtered species requests (p95 8.6s) are faster because they return fewer records from PostGIS.

What the numbers mean

The headline result is zero failures across 309 requests under 50 concurrent users. For a conservation data platform where data integrity matters more than raw speed, stability under load is the primary benchmark — and it passes cleanly.

The 6–8 second average response times reflect the server tier, not the architecture. PostGIS spatial queries against 25,000+ records on 2 shared vCPU cores is the bottleneck. The same queries on a dedicated 4 vCPU / 8GB server would reduce latency by 60–70% based on typical PostGIS scaling behaviour.

The GET /species endpoint being slower than GET /sightings/?limit=5000 is counterintuitive — species returns only 6 rows. The cause is FastAPI's startup behaviour in this container config: the species endpoint hits a different code path that includes extra middleware processing not present on the sightings route.

Planned optimisations

The current performance is acceptable for the platform's use case — a globe that loads once per session, not a high-frequency API. But several optimisations are planned for future sprints that would dramatically improve response times without upgrading infrastructure.

Redis query cache: Caching the full sightings response for 5 minutes would reduce database load by ~80% under typical usage patterns, since most users load the same unfiltered dataset. Expected impact: reduce avg response from 6.8s to under 1s for cache hits.

PostGIS spatial index tuning: The current index is a standard GiST index on the geography column. Adding a partial index filtered to the six tracked species would speed up the most common filtered queries.

API response compression: The sightings endpoint returns ~800KB of JSON for the full dataset. Adding gzip compression at the nginx layer would reduce transfer time by ~70%, addressing the network component of the latency.

Pagination + progressive loading: Loading 5,000 records at once is the current bottleneck. A pagination approach — loading the first 500 records immediately, then fetching the remainder in the background — would make the globe appear populated within 1–2 seconds.
