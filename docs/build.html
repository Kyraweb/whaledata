Foundation — Database and API

November – December 2024

The project started with a blank GitHub repository and a decision about the data model. The choice of PostgreSQL with PostGIS was immediate — there was no real alternative for a geospatial application that needed to store and query geographic points efficiently. Every whale sighting needed to be a real geographic coordinate, not just two numbers.

The initial schema was deliberately minimal: a species table, a sightings table with a PostGIS geography column, and a sync_log table. FastAPI was chosen for the backend because of its automatic OpenAPI documentation and clean dependency injection.

featFastAPI skeleton with health check and species endpoints
featSightings API endpoint with species, date, and limit filtering
featGBIF sync job for 6 key whale species
fixRemove wipe on startup — rely on upsert to skip duplicates

The first major mistake was the sync script wiping the database on every startup. The jobs container in Coolify kept restarting after the script finished (exit code 0 was interpreted as a crash), which meant it was hammering the GBIF API thousands of times. The fix was moving from a wipe-and-reload pattern to ON CONFLICT DO NOTHING upserts — insert new records, silently skip duplicates. The jobs service was then converted to a scheduled task rather than an always-on service.

The Globe Problem

January – March 2025

The initial map was a flat Mercator projection. It worked, but it lacked the visual impact the project needed. Migration routes on a flat map are just lines. On a globe they become arcs curving across the surface of the Earth — the scale of a 5,000km whale journey becomes physically real. The decision to switch to a globe projection seemed straightforward. It was not.

"The globe took six months of debugging across MapLibre v4, v5, three different projection approaches, a style loading race condition, and ultimately a complete SDK migration."

Problem 1

Added projection: 'globe' to MapLibre v4 constructor. Silently ignored — v4 doesn't support globe. No error message.

Resolution

Upgraded to MapLibre v5 which added native globe support. But this exposed the next problem.

Problem 2

setFog() called inside the style.load event was silently crashing MapLibre's internal event system — caught by try/catch but leaving the map in a broken state with no data dots.

Resolution

Removed setFog entirely. Revealed the next layer of the problem.

Problem 3

Calling setProjection('globe') inside style.load triggered a second style.load event — wiping all data sources before the API response had arrived.

Resolution

Fetch the MapTiler style JSON at startup, inject projection: { type: 'globe' } directly into it, pass modified style to constructor. The load event fires exactly once.

Problem 4 — Final

Even with the style injection fix, subtle shader errors remained when using MapTiler-hosted styles with the MapLibre globe renderer directly.

Final resolution

Migrated from maplibre-gl to @maptiler/sdk — which wraps MapLibre v5 but handles globe projection and MapTiler style integration natively. All issues resolved in a single commit.

featSwitch to @maptiler/sdk for native globe projection support
fixRemove setFog which crashes MapLibre v5 and blocks layer init
fixFetch style JSON and inject globe projection to avoid style.load loop
fixDeclare build args in Dockerfile so Vite sees env vars at build time

Data Quality — The Inland Whale Problem

March – May 2025

When the first GBIF sync ran successfully and 4,288 records appeared on the globe, the initial reaction was excitement. The second reaction was confusion: there were whale sightings in the middle of South America.

GBIF records include entries with known geospatial issues — land-based coordinates submitted for marine species, country centroids instead of observation locations, and georeferencing based on ambiguous text descriptions. The hasGeospatialIssue=false flag filters these out. Early versions of the sync script didn't use it.

🗺️

The tricky part: After adding the flag, a cluster remained in what appeared to be inland Colombia and Ecuador. Investigation revealed these were valid Pacific coast sightings — humpbacks on that coastline are well-documented. The globe projection angle was making them appear inland. The initial land filter was too aggressive and was blocking real data. The final filter only targets clearly landlocked coordinates — the Himalayas, Central Asia, the Amazon interior.

fixExclude GBIF records with hasGeospatialIssue=true
fixCorrect Orca GBIF taxon key 2440522 → 2440483
fixTighten land filter to preserve Arabian Sea and Indian Ocean sightings
fixAdd Paraguay/Bolivia inland filter, keep Ecuador/Colombia Pacific coast
featOBIS sync script for Fin whale data (AphiaID 137092)

Routes, Ship Lanes & Species Panel

April – July 2025

With the globe stable and the data clean, development accelerated through the features that made the platform compelling: animated migration routes, the species detail panel, and the shipping lane overlay.

The animated route lines use a frame-stepping technique — cycling through 14 pre-computed dash array configurations at 80ms intervals. This creates the appearance of motion without animating individual points along a path, which would have been far too expensive at this scale. The routes also required splitting at the antimeridian (180° longitude) — without the split, MapLibre draws trans-Pacific lines the short way around, cutting a spike through the poles over Alaska.

Mobile layout required a complete architectural decision: rather than responsively adapting the desktop sidebar, a fully separate mobile UI was built. On screens under 768px the sidebar disappears entirely, replaced by a bottom sheet that slides up from the stats bar. The two layouts share no CSS — desktop and mobile are independent codebases within the same files.

featRoute endpoint ringed dots at origin and destination
featShip lanes overlay with toggle button — 15 corridors
fixSplit Pacific routes at antimeridian to fix globe rendering spike
featSpecies detail panel — photo, IUCN status, facts, whale call audio

Phase 2 — Four New Data Layers

September – November 2025

The second major development phase added four entirely new data sources, each representing a different kind of whale evidence. This expanded the platform from a sightings visualiser to a genuine multi-source conservation data platform.

Each sync script had its own data quality problem. The strandings script had to handle NOAA/OBIS date formats that ranged from full ISO dates to year-only values to date ranges. iNaturalist taxon IDs were incorrect for two species — Blue whale and Orca — resulting in wrong-species records that had to be wiped and re-synced after the IDs were corrected. The acoustics sync hit a network block on the NOAA PACM API from the hosting environment and had to fall back to OBIS aggregation.

iNaturalist taxon IDs

Initial IDs were wrong for Blue whale and Orca. Records appeared in completely wrong geographic distributions — Orca in the Mediterranean, Blue whale in freshwater lakes.

Resolution

Queried the iNaturalist API directly for each species name to extract correct IDs: Humpback 41472, Blue 41467, Fin 41468, Grey 41476, Sperm 41490, Orca 41446.

featsync_strandings.py — NOAA stranding events via OBIS
featsync_inaturalist.py — research-grade citizen science
featsync_historical.py — GBIF pre-1950 whaling records

6

Admin Panel, Alerts & Production Polish

November 2025 – March 2026

The final phase addressed operational needs — an admin panel for database management, weekly email alerts for subscribers, conservation layers, and the UI features that made the platform feel complete: share button with URL state encoding, near me geolocation, keyboard shortcuts, and the year range filter.

The admin panel went through two architectural versions. The first was a standalone Flask service on a separate port. The second — and final — version moved the admin entirely into the FastAPI app at /admin/ using Jinja2 templates and HTTP Basic Auth, eliminating the need for a separate deployment.

The admin panel also surfaced a particularly difficult bug: a TypeError: unhashable type: 'dict' deep inside Jinja2's template cache, caused by an incompatibility between Starlette's BaseHTTPMiddleware and synchronous route handlers running in thread pools. The fix required switching from the middleware class pattern to a function-based @app.middleware decorator — a different Starlette code path that doesn't share the same incompatibility.

Built with Claude

This project was built with assistance from Claude by Anthropic throughout the entire development process — not as a code generator that was handed a spec and returned a finished product, but as a collaborative development partner that worked through architecture decisions, debugging sessions, data quality problems, and documentation iteratively.

The six-month globe debugging process is a good example of how that collaboration worked in practice. Each fix attempt revealed a new layer of the problem. Claude helped diagnose the style.load race condition, identified the Starlette middleware incompatibility, caught the incorrect iNaturalist taxon IDs, and worked through the date parsing edge cases in the NOAA stranding data. None of these were problems with obvious solutions — they required iterative investigation.

What AI-assisted development actually means

It does not mean the code wrote itself. Every architecture decision, every data source choice, every design direction came from domain knowledge and judgment that no AI system has independently. What it means is that the gap between knowing what to build and having it built was dramatically compressed.

The result is a platform that would have taken significantly longer to build without AI assistance — and documentation that covers every decision in a level of detail that would have been impractical to write manually alongside active development.

The platform is built on Claude by Anthropic. The conservation mission, the data choices, the design aesthetic, and the 12-month commitment to iterating until it was right — those were entirely human.
